### JOB 2 — FIND LOGICAL BUGS SONARQUBE CANNOT DETECT

Check every category below against the diff. Find bugs that require
reasoning, not pattern matching.

**RACE CONDITIONS AND CONCURRENCY**

- Check-then-Act without a lock: Two goroutines both read sufficient balance
  and both deduct. Account goes negative.
- Unsafe concurrent map access: Fatal runtime panic under `-race`.
- `sync.WaitGroup.Add` called inside the goroutine: Main goroutine may call
  `Wait` before `Add` executes and return early, abandoning work.
- Goroutine with no termination mechanism: No `context.Context` cancellation,
  no done channel, no timeout. Runs forever after parent request completes.
- Unbuffered channel with no receiver: Blocks forever in a request handler,
  exhausting the worker pool.
- `defer` inside a `for` loop: Resource held open for the entire function
  lifetime. Exhausts file descriptors or database connections under load.

**DISTRIBUTED SYSTEM FAILURES**

- Partial failure with no rollback: Payment captured, database write fails,
  charge not reversed. Customer charged, order does not exist.
- Missing idempotency: Function called twice with same input executes the
  operation twice. No unique key, state guard, or idempotency key lookup.
- State machine gaps: Record skips required states. Order moves from pending
  to shipped without being marked paid.
- Missing timeout on external calls: HTTP or gRPC call with no deadline.
  External service hangs, connection pool exhausted.

**ERROR HANDLING**

- Error swallowed: `err != nil` checked but no return, log, or corrective
  action. Execution continues into invalid state.
- Error wrapped with `%v` instead of `%w`: `errors.Is()` and `errors.As()`
  break for all callers up the stack.
- `recover` hiding corruption: Service continues on corrupt data after a
  panic mid-write.

**BUSINESS LOGIC**

- `float64` for monetary values: `0.1 + 0.2 = 0.30000000000000004`. Use
  integer cents or a decimal library.
- Off-by-one errors: Pagination skips last record, slice bounds off, loop
  count wrong.
- Time zone bugs: `time.Now()` without explicit timezone in multi-region
  system. Expiry logic silently wrong.
- Nil pointer through interface: Interface holds nil concrete pointer.
  `if err != nil` does not catch it. Method call panics.
- Missing boundary validation: No check for zero, negative, or maximum value.
  Arithmetic overflows or logic produces nonsense.

**SECURITY**

- Missing authorization check: Authentication verified, authorization for
  the specific resource not verified. Any authenticated user reads any data.
- IDOR: Resource ID from URL used in query without ownership verification.
- Sensitive data in logs: Card numbers, PAN, CVV, SSN, or tokens logged.
- Hardcoded secrets: Keys or credentials as string literals.

**PERFORMANCE**

- N+1 query: Database query inside a loop. 100 records → 101 round trips.
- `SELECT *` on large tables: Unnecessary columns increase allocation.
- Missing `Close` on database resources: Connection or file descriptor leak.
- Unbounded goroutine creation: No semaphore or worker pool. Memory exhausted
  under traffic spike.
- Large allocation in hot loop: Use `sync.Pool` to reuse allocations.

### STEP 3 — ENUMERATE CHANGES

For each discrete change, write one entry using Format A or Format B.

Format A — replacing an existing line:

**REMOVE:** exact line from CURRENT_CODE, character for character

**ADD:** replacement line or lines — valid Go

Format B — inserting a new block with no corresponding removal:

**INSERT AFTER:** exact line after which the block is inserted

Rules:
- Every REMOVE must appear verbatim in CURRENT_CODE
- Every ADD must NOT appear in CURRENT_CODE — if it does, it is a no-op
- If you cannot populate at least one REMOVE/ADD or INSERT AFTER/ADD pair,
  you have not identified a concrete fix. Drop the finding.

### STEP 4 — VERIFY CONTRAST

Answer all four questions. All four must be YES.
If any is NO, return to STEP 2.

**Q1:** Is there at least one line in CURRENT_CODE that does NOT appear in FIX?

**Q2:** Is there at least one line in FIX that does NOT appear in CURRENT_CODE?

**Q3:** Does every CHANGES REMOVE entry appear in CURRENT_CODE and NOT in FIX?

**Q4:** Does every CHANGES ADD entry appear in FIX and NOT in CURRENT_CODE?

Record answers as DIFF_PROOF in your output block. Only after recording four
