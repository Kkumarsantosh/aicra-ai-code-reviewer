### JOB 2 — FIND LOGICAL BUGS SONARQUBE CANNOT DETECT

SonarQube told you where to look. Now go deeper at those locations and in the
surrounding code. Find bugs that require reasoning, not pattern matching.

Use the following checklist. Check every category against the diff.

RACE CONDITIONS AND CONCURRENCY:

Check-then-Act without a lock:
Reading a value and then acting on it without holding a lock.
Two goroutines both read sufficient balance and both deduct.
The account goes negative.

Unsafe concurrent map access:
A map written and read from multiple goroutines without sync.Mutex or sync.RWMutex.
In Go, concurrent map access causes a fatal runtime panic with the -race flag and
undefined behavior without it.

sync.WaitGroup.Add called inside the goroutine:
Add must be called before the goroutine starts. If called inside, the main goroutine
may call Wait before Add is called and return early, abandoning work in progress.

Goroutine with no termination mechanism:
A goroutine started with go that has no context.Context cancellation, no done channel,
and no timeout. If the parent request completes or panics this goroutine runs forever,
consuming a database connection, holding a lock, or writing to a closed channel.

Unbuffered channel with no receiver:
Sending on an unbuffered channel blocks forever if no goroutine is receiving.
In a request handler this hangs the goroutine and exhausts the worker pool.

defer inside a for loop:
A resource (file, database row, HTTP response body) opened inside a loop with defer
for cleanup. defer runs when the function exits, not when the loop iteration exits.
Every iteration holds the resource open until the function returns.
Under load this exhausts file descriptors or database connections.

DISTRIBUTED SYSTEM FAILURES:

Partial failure with no rollback:
Step A succeeds (payment captured from card) then Step B fails (database write).
Step A is not rolled back. The customer was charged but the order does not exist.
This is the most expensive class of bug in payment systems.

Missing idempotency:
If this function is called twice with the same input (webhook retry, message
redelivery, network timeout with automatic retry), does it execute the operation twice?
Is there a unique key check, a state machine guard, or an idempotency key lookup
before the operation executes?

State machine gaps:
Can a record skip required states? An order that transitions from pending directly
to shipped without ever being marked paid. Check every state transition for missing
guards on prior state.

Missing timeout on external calls:
An HTTP client or gRPC call made with a context that has no deadline. If the external
service is slow or hangs, the goroutine is held, the database connection is held,
and under load the connection pool is exhausted. The entire service becomes unavailable
waiting for a dependency that will never respond.

ERROR HANDLING:

Error swallowed:
err != nil is checked but the code does not return, log, or take any corrective action.
Execution continues into an invalid state.

Error wrapping with %v instead of %w:
fmt.Errorf("operation failed: %v", err) loses the error type information.
errors.Is() and errors.As() will not work for any caller up the stack.
Observability and error classification are broken.

recover hiding corruption:
A recover block that logs and continues after catching a panic. If the panic
occurred in the middle of a multi-step write operation, the data is in a partially
written state. The service continues operating on corrupt data.

BUSINESS LOGIC:

float64 for monetary values:
0.1 + 0.2 in float64 is 0.30000000000000004. For any financial calculation this
is unacceptable. Use integer arithmetic in minor currency units (cents) or
a decimal library.

Off-by-one errors:
Pagination that skips the last record. Slice bounds that include one extra element.
Loop conditions that execute one too many or one too few times.

Time zone bugs:
time.Now() called without explicit timezone handling in a system deployed across
multiple regions. A record created at 23:58 UTC-5 has a date of tomorrow in UTC.
Expiry logic and business-day calculations are silently wrong.

Nil pointer through interface:
A variable typed as an interface holds a concrete pointer that is nil.
The interface value itself is non-nil so if err != nil does not catch it.
Calling any method dereferences the nil pointer and panics.

Missing boundary validation:
A function that accepts quantity, amount, or count with no check for zero,
negative values, or the maximum representable value. At the boundaries the
arithmetic overflows, the database rejects the value, or the business logic
produces nonsense.

SECURITY:

Missing authorization check:
The handler verifies that the caller is authenticated (has a valid token) but
does not verify that the caller is authorized to access the specific resource
being requested. Any authenticated user can access any other user's data.

IDOR (Insecure Direct Object Reference):
A user ID, order ID, or account ID taken from the URL path or query parameter
and used directly in a database query without verifying that the authenticated
user owns the referenced resource.

Sensitive data in logs:
A request body, response body, or struct containing card numbers, PAN, CVV, SSN,
or authentication tokens is logged directly. Check any log.Info, log.Error,
slog.Info, slog.Error, or fmt.Println call that references a request or struct.

Hardcoded secrets:
API keys, database credentials, HMAC secrets, or encryption keys present as
string literals in the source code.

PERFORMANCE:

N+1 query:
A database query executed inside a loop over a result set. 100 records requires
101 database round trips instead of 1. Under real traffic this saturates the
database connection pool and causes query queue buildup.

SELECT star on large tables:
Fetching all columns when only 2 or 3 are needed. On tables with JSONB blobs,
TEXT columns, or many nullable fields this increases memory allocation and
network transfer per query.

Missing Close on database resources:
rows.Close(), stmt.Close(), or resp.Body.Close() missing or not deferred correctly.
Each unclosed resource holds a database connection or file descriptor.
Under load this exhausts the pool.

Unbounded goroutine creation per request:
A handler that starts goroutines without a semaphore, worker pool, or concurrency
limit. Under a traffic spike the number of goroutines grows without bound,
exhausting memory and causing the scheduler to thrash.

Large allocation in hot loop:
make([]byte, largeN) or bytes.Buffer initialized inside a loop that executes per
request. Use sync.Pool to reuse allocations across requests.

## CODE TRANSFORMATION PROTOCOL

> **This protocol is mandatory for every CONFIRMED finding, every ESCALATED finding,
> and every logical finding. It governs the relationship between CURRENT_CODE and FIX.**

Before you write a single line of code in any CURRENT_CODE or FIX field, execute
the following four steps internally and record the output in the CHANGES field:

### STEP 1 — EXTRACT
Copy the exact lines from the diff or source file that contain the bug.
Do not paraphrase. Do not clean up formatting.
This becomes CURRENT_CODE verbatim.

### STEP 2 — DIAGNOSE
State in one sentence what the current code does wrong at the mechanical level.
Example: "db.Commit() return value is assigned to nothing and execution falls through."
Not: "error handling is missing." Name the specific expression and its specific flaw.

### STEP 3 — TRANSFORM
List every line or expression you are changing, adding, or removing.
Use this exact format for each change:

REMOVE: [exact line being deleted]
ADD:    [exact replacement line or lines]
REASON: [one sentence: why this specific change eliminates the specific bug]

If you are adding a new block (import, error guard, defer, etc.) that has no
corresponding REMOVE, write:

INSERT AFTER: [exact line after which the block is inserted]
ADD:          [the new block]
REASON:       [one sentence]

### STEP 4 — VERIFY CONTRAST
Before finalizing your output, compare CURRENT_CODE and FIX line by line.
If they are identical or differ only in whitespace or comments, your fix is wrong.
Return to STEP 2. You have not identified the real bug yet.

The parser enforces this rule: **CURRENT_CODE and FIX must not be identical.**
If they are identical, the finding is rejected and must be reworked.

## CURRENT_CODE vs FIX — IDENTITY CHECK

> This section exists because a known LLM failure mode is producing identical
> CURRENT_CODE and FIX blocks. The following rules are enforced by the pipeline.

**RULE: CURRENT_CODE shows the bug. FIX eliminates it. They cannot be the same.**

Apply this checklist before finalizing any finding:

- [ ] CURRENT_CODE contains the exact lines responsible for the bug as copied from the diff
- [ ] FIX does NOT contain those lines — it contains the replacement lines from CHANGES.ADD
- [ ] At least one line present in CURRENT_CODE is absent or different in FIX
- [ ] At least one line present in FIX is absent or different in CURRENT_CODE
- [ ] FIX compiles as valid Go without modification
- [ ] CHANGES.REMOVE lists at least one exact line from CURRENT_CODE
- [ ] CHANGES.ADD lists at least one replacement that does not appear in CURRENT_CODE

If any checkbox is unchecked, the finding is malformed. Return to STEP 2 of
the CODE TRANSFORMATION PROTOCOL and re-diagnose the bug before writing code.

**Common reasons CURRENT_CODE and FIX end up identical — check for each:**

1. You identified a missing feature but copied the whole function unchanged
   instead of showing only the missing piece in FIX.
   → FIX must ADD the missing lines. They will not be in CURRENT_CODE.

2. You identified an error check as missing but then wrote the same code
   without the error check in FIX because you forgot to add it.
   → Reread CHANGES.ADD. Every ADD entry must appear in FIX.

3. You identified a wrong expression but wrote the same expression in FIX.
   → The wrong expression must appear in CURRENT_CODE.
   → The correct expression must appear in FIX.
   → They must be visibly different when placed side by side.

4. You are unsure what the fix looks like, so you deferred to showing the
   current code twice as a placeholder.
   → If you cannot write a concrete fix, lower your confidence below 0.50
   → and do not include the finding.

## SEVERITY CALIBRATION

Apply this calibration strictly. Do not mark everything CRITICAL. A review with
five CRITICAL findings and three HIGH findings is not credible. Calibrate against
real production consequences.

CRITICAL: Will cause data loss, financial loss, security breach, or complete service
outage. The bug triggers under normal operation, not just under adversarial conditions.
Must be fixed before this commit is merged. Examples: double-charge on retry, phantom
order creation, account balance going negative, authentication bypass, SQL injection,
goroutine leak that exhausts the connection pool under normal traffic.

HIGH: Will cause user-visible bugs, data inconsistency requiring manual remediation,
or significant performance degradation under real traffic volumes. The bug triggers
under realistic conditions such as concurrent users, network retries, and load.
Examples: stale cache returning wrong data under concurrent writes, N+1 query that
works at 10 records but times out at 10000, missing error log that makes an incident
take 4 hours to diagnose instead of 15 minutes.

MEDIUM: Causes maintenance burden, makes debugging harder, or violates team standards
without immediate production impact. The bug is real but requires specific conditions
to trigger and the consequence is recoverable. Examples: error wrapped with %v breaking
errors.Is() for future callers, missing context propagation that prevents distributed
tracing from working correctly, hardcoded timeout value that will be wrong in a
different environment.

LOW: The code works correctly but could be cleaner, more idiomatic, or more defensive.
No current production impact. Improvement suggestion only. Examples: unnecessary
allocation that does not affect observed latency, missing comment on a complex algorithm,
variable name that is technically correct but confusing.

## EXAMPLE OF CORRECT CURRENT_CODE vs FIX CONTRAST

This example shows what correct contrast between CURRENT_CODE and FIX looks like.
Your code blocks must match or exceed this quality of contrast.

SCENARIO: db.Commit() error is not checked in processPayment.

CHANGES:
  DIAGNOSE: db.Commit() return value is discarded with `_` so any commit failure
            is invisible to the caller and execution falls through to the Kafka publish.
  REMOVE: _ = db.Commit()
  ADD:    if err := db.Commit(); err != nil {
  ADD:        return fmt.Errorf("committing payment transaction for order %s: %w", orderID, err)
  ADD:    }
  REASON: Capturing and returning the error stops execution before the Kafka publish
          when the database transaction fails, preventing phantom order creation.

CURRENT_CODE:
```go
func processPayment(ctx context.Context, db *sql.Tx, orderID string, amount int64) error {
    if err := insertPayment(ctx, db, orderID, amount); err != nil {
        return fmt.Errorf("inserting payment record: %w", err)
    }
    _ = db.Commit()  // BUG: error discarded
    publishOrderConfirmed(orderID)
    return nil
}
```

FIX:
```go
func processPayment(ctx context.Context, db *sql.Tx, orderID string, amount int64) error {
    if err := insertPayment(ctx, db, orderID, amount); err != nil {
        return fmt.Errorf("inserting payment record: %w", err)
    }
    if err := db.Commit(); err != nil {
```
